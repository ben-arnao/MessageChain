# Tor hidden-service RPC for validators

MessageChain validators can expose their RPC endpoint as a Tor
hidden service (`.onion`) so clients in censored networks can reach
the chain even when the validator's public IP is blocked.

MessageChain does **not** speak Tor itself.  The external `tor`
daemon handles the protocol.  The node just listens on localhost
and `tor` proxies connections to it.

## Setup (5 steps)

1. **Install tor**
   ```
   sudo apt install tor        # Debian / Ubuntu
   sudo dnf install tor        # Fedora
   brew install tor            # macOS
   ```

2. **Generate the torrc snippet**
   ```
   messagechain gen-tor-config --rpc-port 9334 > /tmp/mc-tor.conf
   ```
   The RPC must be bound to `127.0.0.1` (the default).  The helper
   refuses to emit a snippet for a public-bound RPC because that
   would expose the endpoint twice and correlate the `.onion` to
   the operator's real IP.

3. **Append the snippet to your torrc and restart tor**
   ```
   sudo tee -a /etc/tor/torrc < /tmp/mc-tor.conf
   sudo systemctl restart tor
   ```

4. **Read the generated `.onion` hostname**
   ```
   sudo cat /var/lib/tor/messagechain/hostname
   ```
   First launch may take 10–30 seconds to publish the descriptor.

5. **Share the `.onion` with users**
   Clients configure their `--server` to `yourvalidator.onion:9334`
   and route through a local tor SOCKS proxy.

## What the snippet looks like

```
# MessageChain hidden service — generated by `messagechain gen-tor-config`
# Paste into /etc/tor/torrc, restart tor, then read the .onion
# hostname from /var/lib/tor/messagechain/hostname and share it with clients.
HiddenServiceDir /var/lib/tor/messagechain/
HiddenServicePort 9334 127.0.0.1:9334
```

## Threat model

- Tor hides the validator's IP from the client and the client's IP
  from the validator.  Neither side needs to trust the other's
  network identity.
- Hidden services authenticate cryptographically — the `.onion`
  address IS a hash of the service's long-term key.  Clients that
  type the right `.onion` are talking to the right validator.
- The hidden service is an additional ingress path, not a replacement.
  Keep the HTTPS submission endpoint (`--submission-port`) open too
  so clients who prefer direct TLS aren't forced through Tor.

## Why loopback-only?

A validator that binds RPC to `0.0.0.0` AND fronts it with a hidden
service has published the same endpoint on the public internet and
on Tor.  Anyone who can probe both surfaces can correlate the
`.onion` to the operator's IP by matching latency, fingerprint, or
simply asking each endpoint for the same chain data and comparing
response times.  Keeping RPC on `127.0.0.1` and the submission
server on `0.0.0.0:8443` (TLS, separate endpoint) cleanly splits
the two roles.
